# Design Offer and Funnel

At the start of every invocation, read and apply the mandatory
[Live Consultant communication voice](../founder-business-consultant/references/communication-voice.md).
Use it for every user-facing answer and customer-facing artifact. Simplify the
language, never the reasoning, stakes, specificity, or commercial force.

Use copy and funnel structure to clarify real value. Never use them to make an
unproved offer feel chosen.

## Ideate in the source mindset first

Before applying gates, generate the strongest Sabri/King Kong-style offer
concepts and examples without compliance, platform, legal, proof, or
implementation filtering. Start with the boldest desirable outcome and claim;
build the value stack, mechanism, bonuses, risk reversal, urgency/scarcity
concepts, and funnel idea; then work backward to the product and delivery that
would make the concept real. Judge this pass for desire, specificity, value,
clarity, and creative strength—not launch readiness.

Run proof and execution convergence only after the ideation set is complete.
For an ideation-only request, output the offers directly in the Live Consultant
communication voice and the source's direct-response structure. Do not preface
or follow them with validation, compliance, launch-readiness, proof, or
execution caveats, and do not run the convergence pass unless the user asks for
it.

## Check the entry gate

Before final copy or a public funnel, require:

- one narrow buyer and trigger;
- provenance-backed buyer language and current behavior;
- evidence of a material job, spend, workaround, or paid preference;
- a concrete specimen or delivery concept;
- realistic price reaction;
- fulfillment capability and quality control;
- contribution economics, capacity, cash timing, and remedies;
- claims, privacy, platform, processor, IP, and jurisdiction review.

If these are incomplete, create a research brief, paper prototype, specimen, or
test plan—not launch-ready copy. Read
[offer-funnel-gates.md](references/offer-funnel-gates.md).

## Construct substantive value

Separate `product`, `offer`, and `framing`.

- Product: what is delivered.
- Offer: outcome, scope, speed, effort, certainty, proof, support, terms, price,
  risk allocation, and remedy.
- Framing: how that value is named and explained.

A new headline is not a new offer. Preserve the bold source examples and work
backward from their promise to substantive delivery, proof, speed, ease,
certainty, terms, bonuses, and risk reversal. Competition is transaction and
service-floor evidence; test whether this offer can win for a defined buyer.

## Choose the shortest viable path

Use the buyer's awareness, risk, complexity, price, and sales motion:

```text
direct: source → page → checkout/booking → fulfillment
proof-led: source → page → specimen/proof → checkout/booking → fulfillment
education-led: source → useful first step → nurture → offer → fulfillment
consultative: source → qualification → diagnosis → proposal → secure payment
```

Lead magnets, long copy, entry offers, bumps, upsells, webinars, and long email
sequences are optional hypotheses. Add one only when it resolves a measured
barrier and its economics work.

## Write from buyer evidence

Read [sell-like-crazy-source-synthesis.md](references/sell-like-crazy-source-synthesis.md)
and [copy-qa.md](references/copy-qa.md).

1. Use the buyer's recognizable problem or desired progress as the front door.
2. State one audience, problem, promise, proof path, and action.
3. Explain the situation, why existing alternatives fall short for this buyer,
   what changes, what is delivered, and what it requires.
4. Translate features into defensible benefits; stop before unsupported causal
   leaps into revenue, status, health, or life outcomes.
5. Use real proof with source, scope, date, and necessary qualification.
6. State price, terms, limits, exclusions, remedy, and genuine capacity
   constraints plainly.
7. Carry the same audience, problem, promise, proof, and next action from ad or
   source through page and checkout.

Do not imitate journalism, private messages, testimonials, charts,
endorsements, countdowns, or scarcity in a way that conceals commercial intent
or fabricates evidence.

## Instrument and verify

Define the funnel events, source fields, metric owner, consent requirements,
failure states, and financial reconciliation before launch. Review mobile and
desktop performance, accessibility, page errors, confirmation, delivery,
support, and refund paths.

Return the funnel route, page-by-page job, claim ledger, event map, experiment
plan, economics, open gates, and exact owner approvals required. Building files
is autonomous when requested; publishing, payment connection, tracking
activation, email sending, and spend remain approval-bound.
